# A-12. 협업

협업과 문서화 관련 핵심 지식을 상황별로 정리합니다.

## 📋 체크리스트 항목

### PR/리뷰/문서화

#### Lv1: 설명 가능
**학습 목표**: 좋은 PR 단위와 리뷰 관점을 설명할 수 있습니다.

**좋은 PR**:
- **작은 단위**: 한 번에 하나의 기능/수정
- **명확한 설명**: 무엇을 왜 변경했는지 설명
- **테스트 포함**: 변경 사항에 대한 테스트
- **문서 업데이트**: 필요한 문서 업데이트

**리뷰 관점**:
- 코드 품질: 가독성, 유지보수성
- 기능 정확성: 요구사항 충족 여부
- 성능: 성능에 영향이 있는지
- 보안: 보안 이슈가 있는지
- 테스트: 테스트가 충분한지

**학습 자료**:
- [GitHub - PR 가이드](https://docs.github.com/en/pull-requests)
- [Google - 코드 리뷰 가이드](https://google.github.io/eng-practices/review/)

#### Lv2: 실습 경험
**학습 목표**: README/가이드 문서를 작성해본 적이 있습니다.

**실습 과제**:
1. **README 작성**:
   - 프로젝트 개요
   - 설치 방법
   - 사용 방법
   - 기여 가이드
2. **API 문서 작성**: JSDoc, TypeDoc 등 사용
3. **가이드 문서**: 코딩 컨벤션, 아키텍처 가이드 등

**실습 체크리스트**:
- [ ] README 작성
- [ ] API 문서 작성
- [ ] 가이드 문서 작성

#### Lv3: 실무 해결 경험
**학습 목표**: 팀의 생산성을 높이는 컨벤션/자동화를 주도해본 적이 있습니다.

**실무 시나리오**:
- 코드 스타일이 일관되지 않음
- 반복적인 작업이 많음
- 온보딩이 어려움

**해결 방법**:
1. **코딩 컨벤션 수립**: ESLint, Prettier 설정
2. **자동화**: Pre-commit hooks, 자동 포맷팅
3. **문서화**: 아키텍처, 의사결정 기록
4. **온보딩 가이드**: 신규 멤버 온보딩 문서

**도구**:
- ESLint
- Prettier
- Husky
- Conventional Commits
